# 星夜营地(FightingToStudy)设计规格

- 日期:2026-06-11
- 状态:已与用户逐节确认(产品设计 / 页面布局 / 技术架构)
- 下一步:依本规格制定实现计划(writing-plans)

## 1. 项目目标

做一个本地运行的网页应用,让用户**主动想去学习**,并从中体会到乐趣——像游戏一样"想玩、上瘾"。学习行为本身发生在网页之外(看书、写代码、刷题等);网页是**游戏化的动力系统**:负责让人想开始、能坚持、看得见成长。

### 1.1 用户痛点(四件套)

1. 开始不了(拖延)
2. 坚持不了(三分钟热度)
3. 容易被干扰(注意力被手机/游戏抢走)
4. 没成就感(看不到进步,提不起劲)

### 1.2 成功标准

- 行为指标:连续 14 天中,至少 10 天有 ≥1 次完成的专注委托。
- 主观指标:打开网页时有期待感("看看蛋孵出来没有""今天掉什么")而非负担感。

## 2. 产品设计

### 2.1 世界观:「星夜营地」

用户是一名冒险者,在世界边缘的星空下拥有一座营地。每次专注学习 = 一次**出击委托**;归来结算经验、金币、材料,概率掉落**蛋**。蛋孵化成**生物伙伴**留在营地;材料用于**建造营地建筑**,营地从一堆篝火逐渐变成灯火温暖的家园。

概念融合:RPG 升级流为骨架(委托/经验/等级/随机掉落/稀有度),大量吸收养成收集流(孵蛋/伙伴/图鉴),吸收经营建造流(材料/建筑/营地成长)。

### 2.2 核心循环

```
打开网页 → 今日委托已就绪(3 个,每日刷新,含 1 个 5 分钟低门槛委托)
   ↓ 点「出发」
专注计时(默认 25 分钟,5~120 可调)→ 全屏"冒险中"场景,伙伴陪伴
   ↓ 时间到
结算:EXP + 金币 + 材料(必得)+ 蛋(概率,有保底)→ 奖励逐项动画弹出
   ↓
回营地:孵蛋进度 +1、图鉴点亮、建造升级建筑、升级解锁称号
```

中途撤退 = 本次委托失败、无任何掉落(损失厌恶),但**绝不扣减已有资产**。

### 2.3 痛点 → 机制映射

| 痛点 | 机制 |
|------|------|
| 开始不了 | 打开即见 3 个就绪委托,一键出发零配置;每日固定含 5 分钟「营地散步」超低门槛委托;孵化进度条制造"差一次就孵出来"的钩子 |
| 坚持不了 | **不做惩罚性断签**;资产永续(营地/伙伴/等级一直在);离开 ≥3 天后回归,首页展示"伙伴想你了"+ 回归礼包(金币 200 + 随机材料 ×5),把负罪感变成欢迎感 |
| 易被干扰 | 专注模式全屏沉浸场景;中断则本次奖励泊汤 |
| 没成就感 | 四层即时反馈:数值(EXP/金币)→ 惊喜(随机掉落+稀有度光效)→ 收集(图鉴点亮)→ 空间(营地视觉成长);统计页量化累计成果 |

"不断有新意"的供给:随机掉落与稀有度、24 个待解锁物种、建筑解锁链、等级称号、白天/夜间双场景;v2 可接入 AI 生成每日剧情。

### 2.4 游戏系统(v1)

#### 角色
- 属性:等级、EXP、金币。
- 升级曲线:升至下一级所需 EXP = `100 + (level - 1) × 50`(线性,前期升得快)。
- 称号:每 5 级一档(如 Lv1 见习学者 → Lv5 青铜探索者 → …),共预设 10 档,存于内容配置。

#### 委托(quests)
- 每日委托:每天 3 个,首次访问当天 `GET /api/state` 时按模板池懒生成(无定时任务):
  - 1 × 5 分钟「营地散步」(启动专用,低奖励)
  - 1 × 25 分钟标准委托
  - 1 × 45 分钟深潜委托
  - 标题从模板池随机抽取保持新鲜感;当天未完成的次日标记过期。
- 自建委托:标题(≤30 字)+ 时长(5–120 分钟)+ 可选学科标签(自由文本,带常用建议)。完成后可一键「再来一次」(复制生成新委托)。
- 状态机:`ready → active → done | failed`;每日委托另有 `expired`。

#### 专注会话(sessions)
- 出发时后端创建 session,记录 `started_at`、`ends_at`——**计时真相在后端**,前端只是显示。
- 刷新/误关页面:重新打开时若存在 running session 且未到点 → 自动回到冒险画面继续;已过点 → 显示"凯旋归来"直接进结算。
- 关页期间计时照常进行(个人诚信工具,不做防作弊)。
- 同时只允许一个 active session,后端拒绝并发出击。
- 完成校验:`now ≥ ends_at - 30s`(容忍时钟偏差)。
- 结算幂等:settlement 结果 JSON 存在 session 行上,重复调用返回已存结果,不重复发奖。

#### 掉落(结算时计算)
- EXP = 分钟数 × 2 × (1 + 建筑加成);金币 = 分钟数 × 1 × (1 + 建筑加成)。
- 材料必得:数量 = `1 + floor(分钟 / 15)`,种类按权重随机(木材/石料/星尘/知识水晶,水晶最稀有)。
- 蛋概率掉落:`P = min(0.15 + 分钟 × 0.002, 0.35)`;**保底:若连续 4 次完成未掉蛋,第 5 次必掉**(`pity_counter ≥ 4` 时强制掉落,掉蛋后计数器清零)。
- 稀有度(蛋与物种):普通 70 / 稀有 22 / 史诗 7 / 传说 1(权重)。
- 结算返回**事件序列**,前端按序播放动画:`[{type:'exp'},{type:'gold'},{type:'levelup'?},{type:'drop'...},{type:'hatch'?}]`。

#### 孵化与图鉴
- 蛋进入**孵化队列,一次只孵最老的一颗**(制造期待,防止批量孵化稀释惊喜):每完成一次专注,该蛋进度 +1。当次结算中新获得的蛋从**下一次**专注开始累计进度(同一次结算不既得蛋又长进度)。
- 孵化所需专注次数:普通 3 / 稀有 5 / 史诗 8 / 传说 12。
- 进度达标后,在当次结算事件序列中触发"孵化!"——随机抽取该稀有度下的物种,**未收集者优先**;若该稀有度物种已集齐,允许孵出重复物种(图鉴不变,营地多一只伙伴)。
- 物种首批 24 种:普通 10 / 稀有 8 / 史诗 4 / 传说 2,以 emoji + 名字 + 一句风味文案呈现,定义在内容配置 JSON。
- 图鉴页:网格展示,未收集为剪影(❓+ 暗色),已收集点亮;显示收集进度 n/24。
- 已孵化伙伴出现在营地场景中(emoji 漂浮/走动动画)。

#### 营地建造
- 8 个固定地块;篝火为初始建筑(0 号位,不可拆)。
- 建筑链(示例,完整表在内容配置):帐篷、图书角(EXP 加成)、工坊(材料掉落加成)、星象台(蛋概率加成)、宝物库(金币加成)、瞭望塔、药草园。每栋 1–3 级,消耗木材/石料/水晶。
- 点击空地块弹出可建列表(显示消耗与效果),点击已有建筑可升级。
- 被动加成全局生效,数值小而明确(如每级 +5%)。

#### 统计页
- 总专注时长、总完成次数、当前等级与称号。
- 本周每日专注分钟柱状图(纯 CSS 实现,不引图表库)。
- 学科标签分布(横向条形,纯 CSS)。
- 图鉴收集进度、营地建筑数。

#### 音效(轻量)
- 3 个关键音效:结算完成、稀有掉落、升级;CC0 素材;设置中可关,默认开、音量低。

### 2.5 明确推迟(v1.5 / v2)

- **Boss 战**(大目标 = 讨伐,如"啃完《操作系统》= 20 小时血条",相关专注扣血,击败大奖励)——核心循环验证后第一个加。
- AI 生成每日剧情/随机事件、成就系统、营地装饰自由摆放、多角色、手机端深度优化、数据导出。

## 3. 页面与视觉

### 3.1 页面结构(4 页 + 全局)

| 页面 | 内容 |
|------|------|
| 营地(主页) | 顶栏(角色/等级/EXP 条/资源)+ **左右分栏**:左营地场景(约 58%),右今日委托列表(含预期奖励明细)+ 自建入口;底部导航 |
| 冒险中(专注) | 全屏场景 + 中央计时环 + 当前委托名 + 陪伴伙伴 + 低调的「中途撤退」 |
| 图鉴 | 物种网格(剪影/点亮)、材料一览 |
| 统计 | 累计数据 + 本周柱状图 + 学科分布 |

- 结算以**全屏遮罩层**形式出现在冒险页之上,不算独立路由页。
- 窄窗口(<900px)时左右分栏响应式降级为上下结构。
- 建造交互发生在营地场景内(点地块),不单独成页。

### 3.2 视觉风格:「星夜营地」(治愈 × 暗色魔幻,炫酷但配色克制)

- **夜间主题**(18:00–6:00):墨蓝/碳黑底(#0f1626 一族),暖金(#d8b36a)、雾紫(#8d7bb8)、苔绿点缀;发光效果只用于关键时刻(计时环、稀有掉落、升级),低饱和不刺眼。
- **日间主题**(6:00–18:00):暖沙/奶油底,柔和天蓝,同一布局仅切 CSS 变量。
- 按本地时间自动切换;设置可锁定(自动/日间/夜间),存 localStorage。
- 卡片圆润(圆角 10–16px),emoji 作为图标与生物形象的主要载体;v1 不引入美术素材,场景用分层 CSS 渐变 + emoji + CSS 动画(星星闪烁、篝火光晕脉动、伙伴漂浮)表现。
- 稀有度配色:普通灰白 / 稀有蓝紫 / 史诗深紫 / 传说暖金,光效强度随稀有度递增。

## 4. 技术架构

### 4.1 总体结构

```
FightingToStudy/
├── package.json          # npm workspaces 或根脚本(dev/start/build/test)
├── client/               # React 18 + Vite
│   └── src/
│       ├── pages/        # Camp, Adventure, Collection, Stats
│       ├── components/   # QuestCard, TimerRing, RewardSequence, CampScene, ...
│       ├── api/          # fetch 封装 + useGameState hook
│       ├── theme/        # 日/夜 CSS 变量
│       └── assets/       # 音效等
├── server/               # Node.js + Express
│   └── src/
│       ├── routes/       # REST 路由
│       ├── services/     # 结算、掉落、每日委托生成、孵化(纯函数为主)
│       ├── db/           # better-sqlite3 初始化、迁移、备份
│       └── content/      # species.json, buildings.json, quest-templates.json, titles.json
└── data/                 # app.db + backups/(gitignored)
```

- 技术选型:React 18、Vite、react-router-dom、framer-motion(奖励序列动画)、Express、better-sqlite3、Vitest(+ supertest、React Testing Library)。不引入状态管理库与图表库。
- **日常使用**:`npm start` → Express(端口 3001)托管打包后前端 + API,浏览器打开 `http://localhost:3001`。
- **开发**:`npm run dev` → Vite(5173,proxy `/api` → 3001)+ 后端并行。
- 内容(物种/建筑/委托模板/称号)与存档(SQLite)分离:加新内容只改 JSON,不动数据库。

### 4.2 数据模型(SQLite)

日期时间一律 ISO 8601 文本(UTC);"游戏日"用本地日期 `YYYY-MM-DD` 单独存。

```sql
player(id=1, name, level, exp, gold, pity_counter, last_completed_at, created_at)
quests(id, title, type 'daily'|'custom', duration_min, subject_tag,
       status 'ready'|'active'|'done'|'failed'|'expired', daily_date, created_at)
sessions(id, quest_id, started_at, ends_at,
         status 'running'|'completed'|'abandoned',
         settlement_json, completed_at)
inventory(id, item_key, qty)                  -- 材料库存
eggs(id, rarity, progress, required, status 'incubating'|'hatched', obtained_at)
creatures(id, species_key, rarity, hatched_at)
buildings(id, slot_index 0..7, building_key, level, built_at)
settings(key, value)                          -- 预留
```

### 4.3 API

| 方法与路径 | 说明 |
|------|------|
| `GET /api/state` | 聚合首屏:player、今日委托(懒生成)、running session、营地、资源、孵化中蛋、回归礼包事件 |
| `POST /api/quests` | 自建委托 `{title, duration_min, subject_tag?}` |
| `POST /api/quests/:id/start` | 出发:校验无并发 session → 创建 session,返回 `{session_id, ends_at}` |
| `POST /api/sessions/:id/complete` | 到点结算:时间校验 → 计算奖励 → 写库 → 返回事件序列;幂等 |
| `POST /api/sessions/:id/abandon` | 撤退:标记 failed/abandoned,无掉落 |
| `GET /api/collection` | 物种目录 × 收集状态 |
| `GET /api/stats` | 统计数据 |
| `POST /api/buildings` | 建造/升级 `{slot_index, building_key}`,校验材料并扣减 |

### 4.4 错误处理

- 后端:统一错误中间件返回 `{error: message}` + 合适状态码;入参手动校验(端点少,不引校验库);掉落计算使用可注入种子的 RNG(生产用随机种子,测试用固定种子)。
- 前端:结算请求失败时**保留结算现场**并显示「重试」按钮,奖励绝不静默丢失;其余请求失败 toast 提示("信号不佳");API 封装统一处理。
- 数据安全:服务启动时自动备份 `app.db` → `data/backups/app-<timestamp>.db`,滚动保留最近 3 份。

### 4.5 测试策略

- **后端为重**(Vitest + supertest):
  - 结算数值:EXP/金币/材料公式、建筑加成叠加;
  - 掉落:概率边界(种子 RNG 确定性断言)、保底第 5 次必掉、稀有度权重;
  - 孵化:队列只孵最老、进度达标触发、未收集优先;
  - session 状态机:start→complete/abandon、并发拒绝、过点恢复、**结算幂等**;
  - 每日委托:懒生成当天仅一次、过期标记。
- **前端轻量**(Vitest + RTL):计时器倒计时与恢复逻辑、结算事件序列按序渲染。
- 验收:手动走通"出发 → 结算 → 孵化 → 建造"主流程;可选 Playwright 固化该链路。

## 5. 实现优先级(供计划阶段参考)

1. 骨架:仓库结构、Express + SQLite、`GET /api/state`、React 路由与主题变量;
2. 核心循环:委托 → 出发 → 计时 → 结算(数值与事件序列,先无动画);
3. 掉落与孵化:材料/蛋/保底/图鉴;
4. 结算动画与音效(framer-motion 事件序列);
5. 营地场景与建造(日/夜主题、CSS 场景、地块交互);
6. 统计页、回归礼包、备份、打磨。
